Conflict Resolution Demo

VERSA (CRDT)

  • Client A edits field
  • Client B edits same field
  • Both diffs sent to relay
  • CRDT merges automatically
  • Both clients converge

REST (optimistic lock)

  • Client A fetches (GET)
  • Client B fetches (GET)
  • Client A saves (PUT, v1)
  • Client B saves (PUT, v1) — 409
  • Client B re-fetches + retries

GraphQL subscription

  • Client A mutates
  • Server broadcasts full object
  • Client B mutates (stale base)
  • Server broadcasts again
  • Last-write-wins, A's edit lost

VERSA latency is measured end-to-end: time from sending a binary delta to receiving it on a second WebSocket connection in the same room. REST latency models 500ms polling: average staleness = poll interval / 2 + network RTT. Bandwidth includes request headers per poll. GraphQL subscription latency adds a server resolver round-trip (~8ms, p50 Postgres SELECT) to the WebSocket RTT, because the subscription handler must re-fetch the full object before broadcasting. Payload is the full JSON object on every update regardless of what changed — a 1-character edit sends the same bytes as creating the record. All three transports are benchmarked against the same relay host (). Source: github.com/camdenslade/VERSA
